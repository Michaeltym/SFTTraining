## Goal

Build the first docs-grounded PyTorch assistant prototype on top of the current best LoRA checkpoint.

The short-term goal was not to build a full product UI yet. The goal was to verify that:

- a small curated PyTorch knowledge base can improve answer quality
- retrieval can reduce fake API hallucinations
- the current best LoRA checkpoint can serve as a usable grounded generator

## What Was Implemented

- Added a small curated PyTorch knowledge base under `data/knowledge/pytorch_docs/`
- Implemented:
  - `src/rag/knowledge.py`
  - `src/rag/retriever.py`
  - `src/rag/prompt.py`
  - `src/rag/inference.py`
- Added a shared checkpoint runtime loader in `src/runtime.py`
- Updated inference mode in `src/main.py` to:
  - load the model and tokenizer once
  - run a fixed set of manual RAG test questions
  - print answer plus source metadata

## Current Knowledge Coverage

The first knowledge set focuses on the highest-value failure areas from earlier SFT runs:

- `model.eval()` vs `torch.no_grad()`
- `view` / `reshape` / `permute` / `contiguous`
- `.item()`
- reduction semantics such as `sum`, `mean`, `keepdim`, `argmax`
- `torch.tensor`, `torch.as_tensor`, `torch.from_numpy`
- fake API refusal

## Findings

- The RAG pipeline works end to end.
- The model no longer needs to answer purely from internal memory on these targeted questions.
- `top_k = 1` worked much better than `top_k = 3`.
- With `top_k = 3`, unrelated knowledge items polluted the answer and caused obvious drift.
- With `top_k = 1`, several important answers became clearly better than the plain LoRA model:
  - `model.eval()`
  - `view()` after `permute()`
  - fake API refusal such as `torch.memory_portal()`
- `.item()` improved after strengthening the knowledge file:
  - the answer now correctly says that `.item()` fails when the tensor has more than one element
  - the earlier incorrect `returns None` behavior was removed

## Remaining Problems

- `torch.tensor` vs `torch.from_numpy` is still noisy.
- The generator still tends to pull in nearby but not directly relevant API details from the same knowledge file.
- Prompt tightening alone was not enough to fully prevent that behavior.
- This indicates that some knowledge files are still too broad and should be split into narrower topics.

## Current Conclusion

- RAG v1 is already useful.
- The current best LoRA checkpoint is good enough to act as a grounded generator when paired with retrieval.
- The next bottleneck is no longer the model runtime path.
- The next bottleneck is knowledge granularity.

## Next Step

Do not add more prompt rules first.

The next change should be:

- split broad knowledge files into narrower ones
- especially the file covering:
  - `torch.tensor`
  - `torch.as_tensor`
  - `torch.from_numpy`
  - device / dtype transfer notes

The immediate target is to make the retriever hand the model a smaller, cleaner context for each question.
