## Goal

Build the first docs-grounded PyTorch assistant prototype on top of the current best LoRA checkpoint.

The short-term goal was not to build a full product UI yet. The goal was to verify that:

- a small curated PyTorch knowledge base can improve answer quality
- retrieval can reduce fake API hallucinations
- the current best LoRA checkpoint can serve as a usable grounded generator

## What Was Implemented

- Added a small curated PyTorch knowledge base under `data/knowledge/pytorch_docs/`
- Implemented:
  - `src/rag/knowledge.py`
  - `src/rag/retriever.py`
  - `src/rag/prompt.py`
  - `src/rag/inference.py`
- Added a shared checkpoint runtime loader in `src/runtime.py`
- Updated inference mode in `src/main.py` to:
  - load the model and tokenizer once
  - run a fixed set of manual RAG test questions
  - print answer plus source metadata

## Current Knowledge Coverage

The first knowledge set focuses on the highest-value failure areas from earlier SFT runs:

- `model.eval()` vs `torch.no_grad()`
- `view` / `reshape` / `permute` / `contiguous`
- `.item()`
- reduction semantics such as `sum`, `mean`, `keepdim`, `argmax`
- `torch.tensor`, `torch.as_tensor`, `torch.from_numpy`
- fake API refusal

## Findings

- The RAG pipeline works end to end.
- The model no longer needs to answer purely from internal memory on these targeted questions.
- `top_k = 1` worked much better than `top_k = 3`.
- With `top_k = 3`, unrelated knowledge items polluted the answer and caused obvious drift.
- With `top_k = 1`, several important answers became clearly better than the plain LoRA model:
  - `model.eval()`
  - `view()` after `permute()`
  - fake API refusal such as `torch.memory_portal()`
- `.item()` improved after strengthening the knowledge file:
  - the answer now correctly says that `.item()` fails when the tensor has more than one element
  - the earlier incorrect `returns None` behavior was removed
- Splitting broad knowledge files helped:
  - the earlier `tensor_creation_and_numpy_interop` file was too broad
  - splitting it into narrower files reduced cross-topic contamination
- Splitting fake API refusal knowledge also helped:
  - moving from one mixed fake-API file to one general rule file plus a few small example files reduced cross-example pollution
  - `torch.memory_portal()` and `nn.SuperLayer` became cleaner refusal cases
- The first saved RAG evaluation result confirms that several high-value questions are now stable:
  - `model.eval()` was answered correctly
  - `view()` after `permute()` was answered with the correct non-contiguous explanation, though the answer was slightly truncated
  - `.item()` on multi-element tensors was answered correctly
  - fake API refusal for `torch.memory_portal()`, `torch.quantum_backprop()`, and `nn.SuperLayer` was correct and concise
- Prompt tightening helped more than generation penalties:
  - keeping `top_k = 1`
  - using a short constrained prompt
  - keeping answer scope narrow
  produced better results than broader context plus more rules

## Remaining Problems

- The generator can still produce truncation or short incomplete endings on some answers.
- `torch.cat` vs `torch.stack` still tends to over-continue after the main contrast is already correct.
- `view` vs `reshape` still tends to repeat the same idea and end abruptly.
- This means the current bottleneck is no longer mostly retrieval quality; it is answer stability after the correct knowledge has already been retrieved.

## Current Conclusion

- RAG v1 is already useful.
- The current best LoRA checkpoint is good enough to act as a grounded generator when paired with retrieval.
- The next bottleneck is no longer the model runtime path.
- Knowledge granularity was a real issue and improved after splitting broad files.
- The remaining bottleneck is generation reliability on high-precision comparison questions.
- Compared with plain LoRA, RAG is now clearly better on targeted refusal and PyTorch-semantics questions that are covered by the curated knowledge base.
